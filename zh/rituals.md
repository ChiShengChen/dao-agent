# 醮典科儀大全 ── Multi-Agent Workflow Patterns

多位神將協壇行法時，科儀編排決定成敗。以下為經過驗證的經典科儀陣式。

---

## 科儀壹：步罡踏斗（Pipeline）

取自道教「步罡踏斗」之法——法師按北斗星位依序踏步，每一步銜接下一步，不可亂序。

```
[神將甲] → [神將乙] → [神將丙] → 功德圓滿
```

**適用道場**：法事有明確先後，前壇功德是後壇供物。

**典型法事**：
- 資料煉化：採集 → 淨化 → 轉化 → 入庫
- 程式碼修煉：生成 → 審查 → 試煉 → 部署
- 法典處理：提取 → 翻譯 → 排版

**行法要訣**：
- 各壇之間以法物（檔案）傳遞，約定統一的中間法器格式
- 每壇圓滿後將功德寫入約定壇場，下一壇從該壇場取用
- 每壇法物皆保留不消，方便事後追溯

**大法主敕令片段**：
```
依北斗步位，依序行法：
1. 敕令 [神將甲] 啟壇，功德存於 /tmp/gangdou/step1/
2. 確認 /tmp/gangdou/step1/ 中有預期法物
3. 敕令 [神將乙]，取 /tmp/gangdou/step1/ 為供物，功德存於 /tmp/gangdou/step2/
4. 確認 /tmp/gangdou/step2/ 中有預期法物
5. 敕令 [神將丙]，取 /tmp/gangdou/step2/ 為供物，功德存於 /tmp/gangdou/final/
任何一壇遭天劫，即時停止科儀，回報劫難壇位與劫因。
```

## 科儀貳：五雷正法（Fan-out / Fan-in）

取自道教五雷正法——法師一聲令下，五方雷神同時響應，各司其職，最後匯聚天雷一擊。

```
             ┌→ [雷神甲] →┐
[先鋒將] →  ├→ [雷神乙] →┤  → [匯雷真人]（匯總）
             └→ [雷神丙] →┘
```

**適用道場**：一件法事可拆為多路並行，最終匯於一壇。

**典型法事**：
- 多語翻譯：一份法典同時譯為多國文字
- 平行試煉：同一法物在多個天地中同時試煉
- 多角推演：同一資料從不同法門分析

**行法要訣**：
- 各路雷神完全獨立，互不知曉彼此
- 每位雷神功德存入各自子壇
- 匯雷真人負責讀取所有子壇並匯總
- 須預設部分雷神渡劫失敗的情況——匯雷真人需能處理「N 路中 M 路成功」

**大法主敕令片段**：
```
1. 敕令 [先鋒將] 備妥供物，存於 /tmp/wulei/input/
2. 同時敕令以下雷神（各路互不相干）：
   - [雷神甲] 取 /tmp/wulei/input/，功德存 /tmp/wulei/a/
   - [雷神乙] 取 /tmp/wulei/input/，功德存 /tmp/wulei/b/
   - [雷神丙] 取 /tmp/wulei/input/，功德存 /tmp/wulei/c/
3. 待五方雷神皆圓滿
4. 敕令 [匯雷真人] 匯總 /tmp/wulei/a/, b/, c/，功德存 /tmp/wulei/final/
```

## 科儀參：九轉金丹（Iterative Refinement）

取自道教煉丹之法——丹藥須經九轉方能大成。一轉不成，再煉；二轉不成，三煉。直到丹成。

```
[造化真人（煉丹）] → [監察御史（試丹）] → 不成？→ 回爐重煉
                                           → 丹成？→ 功德圓滿
```

**適用道場**：需要反覆精煉直到達標。

**典型法事**：
- 程式碼生成 + Code Review 循環
- 文案撰寫 + 編輯修潤
- 設計迭代 + 回饋修正

**行法要訣**：
- 設定最多幾轉（通常三到五轉），避免無限回爐
- 監察御史須出具結構化奏摺（不只「不成」二字，要說明何處不成、為何不成）
- 每轉中間產物皆保留——一轉丹、二轉丹各存一份
- 二轉以後，造化真人收到的是「上一轉丹 + 御史奏摺」

**大法主敕令片段**：
```
最多煉 [N] 轉：
1. 敕令 [造化真人] 煉製/修改法物，存於 /tmp/jindan/v{N}/draft
2. 敕令 [監察御史] 試驗 /tmp/jindan/v{N}/draft
3. 若御史判定丹成（無天劫），將 draft 移至 /tmp/jindan/final/ 並收壇
4. 若御史判定不成，將奏摺與 draft 一同送入下一轉
5. 若達最大轉數仍不成，取最後一轉之丹並附上所有未解之奏摺呈報法師
```

## 科儀肆：守關法（Gatekeeper）

取自道教修煉「過關」之法——每過一關需守關神將驗明正身。

```
[供物] → [守關天王（驗入）] → 通過 → [行法道兵] → [守關天王（驗出）] → 功德
                              → 不通過 → 驅回/報告
```

**適用道場**：法事前後需要安全/品質關卡。

**典型法事**：
- API 輸入驗證 → 處理 → 輸出驗證
- 程式碼提交 → 安全掃描 → 合規檢查
- 資料匯入 → schema 驗證 → 品質檢驗

**行法要訣**：
- 守關天王可以是同一位（入關出關用同一符），也可以分設（入關嚴、出關鬆或反之）
- 驅回時必須說明原因，不可只說「不通過」
- 急件可設「特別通行」條件，但須記入功德簿

## 科儀伍：三清會議（Expert Panel）

取自道教三清——玉清、上清、太清——各代表不同的道法視角，共議天地大事。

```
             ┌→ [玉清仙（視角A）] →┐
[法事] →     ├→ [上清仙（視角B）] →┤  → [太上法主（綜合判斷）] → 最終決策
             └→ [太清仙（視角C）] →┘
```

**適用道場**：同一件法事需要多個角度綜合評估。

**典型法事**：
- 技術選型：效能仙人 + 安全仙人 + 維護仙人分別論道
- 風險推演：財務 + 法律 + 技術各自奏議
- 架構決策：從可擴展性、簡潔性、效能三面論道

**行法要訣**：
- 三清各持同一份法物，但符籙不同（強調不同的推演角度）
- 太上法主須有明確的決斷法則（加權、票決、或其他機制）
- 三清之間互不通氣，只與法主通氣——避免相互影響

## 科儀陸：天地人三才陣（Layered Architecture）

取自道教「天地人三才」——天覆地載人居中。

```
[天層：都天大法主] ── 戰略決策、任務分派
        ↕
[人層：專業神將群] ── 具體執行各項法事
        ↕
[地層：護法基礎] ── 監控、日誌、錯誤處理
```

**適用道場**：大型系統，需要分層管理。

**行法要訣**：
- 天層只做決策和分派，不碰具體法術
- 人層各司其職，互相透過檔案系統溝通
- 地層默默運行，只在異象出現時上報天層
- 三層各有獨立的壇場（目錄），互不混雜

## 科儀選擇指南

| 道場情境 | 推薦科儀 |
|---------|---------|
| 法事有明確階段順序 | 步罡踏斗 |
| 同一法事需多路並行 | 五雷正法 |
| 品質要求高，需反覆精煉 | 九轉金丹 |
| 法事前後需檢查關卡 | 守關法 |
| 需要多角度綜合判斷 | 三清會議 |
| 大型系統分層管理 | 天地人三才 |
| 以上組合 | 巢狀科儀——在科儀節點中嵌入子科儀 |

## 道法根本戒律

1. **壇場是唯一的信道**——神將之間以法物（檔案）溝通，不可試圖以 stdout 或環境變數傳遞複雜法物
2. **各壇互不侵犯**——每位神將的工作壇場（目錄）要隔離，避免法物混亂覆寫
3. **法物不可銷毀**——所有中間產物保留，方便追溯因果
4. **設全壇時限**——整場科儀不應超過合理時辰，道亦有時
5. **先以小壇試法**——用簡單供物走一遍完整科儀，確認法度無誤再放大規模
6. **道法自然**——能用一位神將解決的，不要動用三位；簡單即是大道
