# 傳承譜系 ── Template Inheritance

一脈相承，萬法歸宗。子符繼承母符的共用設定，只覆寫差異部分——減少重複、統一門規。

---

## 傳承之理

在大醮科儀中，多位道兵常共享相同的：
- 壇場設定（working directory）
- 戒律（天條 + 壇律）
- 符文風格（命名慣例、程式語言）
- 結界設定（允許/禁止路徑）
- 法力配置（模型選擇）

與其在每道符籙中重複這些，不如建立一道**母符**（Base Talisman），讓各道兵的符籙作為**子符**繼承之。

---

## 母符格式

```markdown
# 母符：[門派名 / 專案名]

## 壇場
- 根目錄：[path]
- 程式語言：[language]
- 框架：[framework]
- 套件管理：[package manager]

## 門規（共用戒律）
天條：
- TC-001 不得洩露天機
- TC-002 不得越界行法：限 [path] 及其子目錄
- TC-003 不得無中生有
- TC-005 不得吞噬天劫：錯誤記入 [log path]

壇律：
- CS-001：[專案級禁制]
- CS-002：[專案級禁制]

## 門風（共用風格）
- 命名慣例：[camelCase / snake_case / ...]
- 縮排：[spaces / tabs, N]
- 註解語言：[中文 / English]
- 錯誤處理：[模式描述]

## 共用結界
- 允許讀取：[glob patterns]
- 允許寫入：[glob patterns]
- 禁止存取：[glob patterns]

## 共用法力配置
- 預設模型：[model]
- 預設 max_turns：[N]

## 共用加持
- [自省訣 / 通報訣 / ...]
```

---

## 子符格式

子符只寫**差異部分**，其餘繼承母符。

```markdown
# 子符：[法號]
# 承自：[母符名]

## 天命
[此道兵的具體任務——必填，不可繼承]

## 覆寫項目

### 覆寫戒律（追加）
職律：
- RC-XXX：[此道兵的職律]

壇律追加：
- CS-XXX：[額外壇律]

### 覆寫結界（擴展寫入範圍）
- 追加允許寫入：[額外路徑]

### 覆寫法力
- 模型：[若此道兵需要不同模型]
- max_turns：[若此道兵需要不同上限]

## 專屬科儀
[此道兵的具體步驟——不繼承，每位道兵各有各的行法程序]

## 專屬功德簿
[此道兵的輸出規格——不繼承]
```

---

## 繼承規則

| 項目 | 能否繼承 | 說明 |
|------|---------|------|
| 壇場設定 | ✅ 繼承 | 全壇共用 |
| 天條戒律 | ✅ 繼承 | 所有道兵共守 |
| 壇律 | ✅ 繼承 + 可追加 | 子符可追加，不可刪除母符的壇律 |
| 門風 | ✅ 繼承 | 統一風格 |
| 結界 | ✅ 繼承 + 可擴展 | 子符可擴展允許範圍，不可縮小禁止範圍 |
| 法力配置 | ✅ 繼承 + 可覆寫 | 子符可選用不同模型 |
| 加持 | ✅ 繼承 + 可追加 | 子符可追加加持，不可移除母符加持 |
| 天命 | ❌ 不繼承 | 每位道兵天命各異 |
| 科儀步驟 | ❌ 不繼承 | 各有各的行法程序 |
| 功德簿 | ❌ 不繼承 | 各有各的輸出規格 |

**核心原則**：子符可**追加**或**覆寫**，但不可**移除**母符的戒律與禁制（安全只增不減）。

## 完整範例

### 母符

```markdown
# 母符：電商專案

## 壇場
- 根目錄：/workspace/ecommerce
- 程式語言：TypeScript
- 框架：Next.js + Prisma
- 套件管理：pnpm

## 門規
天條：TC-001, TC-002(/workspace/ecommerce), TC-003, TC-005(/workspace/ecommerce/logs/)
壇律：
- CS-001：不得直接修改 /prisma/migrations/ 下的已有 migration
- CS-002：所有 API route 必須有 input validation
- CS-003：不得安裝未經審核的 npm 套件

## 門風
- 命名慣例：camelCase（變數/函式）、PascalCase（型別/元件）
- 縮排：2 spaces
- 錯誤處理：使用 Result<T, Error> pattern

## 共用結界
- 允許讀取：/workspace/ecommerce/**/*
- 允許寫入：/workspace/ecommerce/src/**、/workspace/ecommerce/tests/**
- 禁止存取：/workspace/ecommerce/.env*、/workspace/ecommerce/node_modules/**

## 共用法力
- 預設模型：claude-sonnet-4-6
- 預設 max_turns：10

## 共用加持
- 自省訣
- 通報訣
```

### 子符 A：造化真人

```markdown
# 子符：購物車造化真人
# 承自：電商專案

## 天命
實作購物車的新增商品功能：POST /api/cart/items

## 覆寫項目

### 追加壇律
- CS-004：購物車邏輯須有對應的 unit test

### 追加結界寫入
- 追加允許寫入：/workspace/ecommerce/prisma/schema.prisma

### 覆寫法力
- max_turns：15（此任務較複雜）

## 專屬科儀
1. 讀取現有購物車相關程式碼，理解資料模型
2. 在 Prisma schema 中新增/修改必要的 model
3. 在 /src/app/api/cart/items/ 新增 route handler
4. 在 /src/lib/ 新增購物車業務邏輯
5. 撰寫對應測試
6. 執行 pnpm test 確認通過

## 專屬功德簿
- 新增/修改的檔案清冊
- API endpoint 文件（method, path, request/response schema）
- 測試覆蓋率報告
```

### 子符 B：監察御史

```markdown
# 子符：購物車監察御史
# 承自：電商專案

## 天命
審查購物車造化真人的產出，確保品質達標。

## 覆寫項目

### 追加職律
- RC-002：只審不改

### 覆寫法力
- 模型：claude-opus-4-6（審查需要更強推理）
- max_turns：5

## 專屬科儀
1. 讀取造化真人的功德簿，了解修改範圍
2. 逐項審查：
   - API 輸入驗證是否完備
   - 錯誤處理是否使用 Result pattern
   - 測試是否覆蓋正常路徑、邊界條件、錯誤路徑
   - 命名是否符合門風
   - 是否違反任何壇律
3. 呈上奏摺

## 專屬功德簿
- 監察奏摺（標準格式）
```

## 多層傳承

母符也可以有母符，形成傳承鏈。但最多三層：

```
祖符（公司級）
  └→ 母符（專案級）
      └→ 子符（道兵級）
```

超過三層則過於複雜，建議扁平化。

---

## 生成帶傳承的符籙

在書符時，若法師提供了專案上下文或共用規範，自動：

1. 檢查是否已有可用的母符
2. 若有，生成子符格式（只寫差異）
3. 若無，建議法師先建立母符，再生成子符
4. 在功德簿中標注「承自：[母符名]」
