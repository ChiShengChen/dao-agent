# 進階科儀 ── Advanced Workflow Patterns

在六大基礎科儀之上的進階法門：巢狀科儀、輪值道兵、替身道兵。

---

## 科儀柒：巢狀科儀（Nested Workflows）

大醮之中套小醮，科儀之內嵌科儀。將基礎科儀作為積木自由組合。

### 組合心法

任何科儀的某一壇位，都可以替換為另一場完整的子科儀。例如：

**步罡踏斗 × 九轉金丹：**
```
[先鋒探查] → [九轉金丹（程式碼生成+審查迴圈）] → [部署道兵]
```

**五雷正法 × 守關法：**
```
              ┌→ [守關法（驗入→翻譯A→驗出）] →┐
[備料道兵] → ├→ [守關法（驗入→翻譯B→驗出）] →┤ → [匯總道兵]
              └→ [守關法（驗入→翻譯C→驗出）] →┘
```

**天地人三才 × 三清會議：**
```
[天層：大法主]
    ↓ 分派法事
[人層：三清會議（多角度評估）] → 決策
    ↓ 執行決策
[地層：護法監控]
```

### 巢狀生成法

當使用者描述的需求涉及多層結構時，按以下程序生成：

1. **繪製總圖**——先用 ASCII 或 Mermaid 畫出整體拓撲
2. **標記巢狀點**——哪些壇位需要展開為子科儀
3. **由內而外書符**——先生成最內層的子科儀符籙，再生成外層
4. **定義接口法物**——每層之間的輸入輸出必須明確對接
5. **設定巢狀渡劫**——子科儀渡劫失敗如何上報外層

### 巢狀敕令片段

```
主科儀：步罡踏斗
  壇位一：[千里眼] 探查需求
  壇位二：【子科儀：九轉金丹】
    煉丹者：[造化真人] 依壇位一的探查結果生成程式碼
    試丹者：[監察御史] 審查程式碼
    最多三轉
    子科儀功德存放：/tmp/nested/inner/
  壇位三：[部署道兵] 取 /tmp/nested/inner/final/ 部署

巢狀渡劫：
- 若子科儀三轉未成，將最後一版丹藥與奏摺一併提交壇位三
- 壇位三收到帶奏摺的法物時，僅部署到 staging 環境
```

### 巢狀深度戒律

- **最多三層巢狀**——超過三層則道法混亂，應重新設計
- **每層有獨立壇場**——`/tmp/ritual/L1/`、`/tmp/ritual/L1/sub1/`
- **子科儀對外是黑箱**——外層只看最終功德，不必知道內部幾轉幾壇

## 科儀捌：輪值道兵（Human-in-the-Loop）

行法至關鍵壇位時，暫停等待法師（使用者）親自過目確認，方可繼續。適用於高風險操作。

### 陣式

```
[道兵甲] → ⏸️ 法師確認 → [道兵乙] → ⏸️ 法師確認 → [道兵丙]
```

### 適用道場

- 涉及生產環境部署
- 涉及資料刪除或不可逆操作
- 涉及對外發送（郵件、訊息、API 呼叫）
- 涉及金錢或合約相關操作

### 輪值符文

在需要暫停的壇位，使用以下符文：

```
⏸️ 關口：[關口名稱]

在此處暫停行法，將以下資訊呈報法師：

呈報內容：
- 已完成：[列出前壇成果摘要]
- 即將執行：[列出下一步具體操作]
- 影響範圍：[列出此操作的影響]
- 可回退：[是/否，及回退方式]

等待法師敕令：
- 「准」→ 繼續行法
- 「改」→ 按法師新指示修正後重新呈報
- 「止」→ 收壇，保留當前功德
```

### 大法主敕令片段（含輪值）

```
依序行法，遇關口則暫停：

1. 敕令 [千里眼] 掃描生產環境現狀，功德存 /tmp/deploy/scan/
2. ⏸️ 關口「部署確認」：
   呈報掃描結果與即將部署的版本差異
   等待法師確認
3. 敕令 [部署道兵] 執行部署，功德存 /tmp/deploy/result/
4. 敕令 [護法天王] 監控部署後 5 分鐘的健康指標
5. ⏸️ 關口「回退判斷」：
   呈報健康指標
   若法師判定異常，執行回退科儀
```

### 輪值設計要訣

- **呈報必須充分**——法師不應需要自行查閱才能做判斷
- **預設為「止」**——若法師長時間未回應，不可自行繼續
- **記錄法師決策**——將每次確認/拒絕記入功德簿，方便事後追溯
- **關口不可跳過**——即使前壇一切順利，該停的地方必須停

## 科儀玖：替身道兵（Fallback Agent）

某位神將渡劫失敗時，自動啟動一位替身道兵，以不同策略重試。

### 陣式

```
[主將] → 成功 → 繼續
    │
    └→ 渡劫失敗 → [替身道兵（策略B）] → 成功 → 繼續
                        │
                        └→ 渡劫失敗 → [末路道兵（最簡策略）] → 成功/呈報
```

### 適用道場

- 主要策略可能因環境差異而失敗（例如某工具未安裝）
- 需要優雅降級（先嘗試最佳方案，失敗則退而求其次）
- 外部依賴不穩定（API 可能超時、服務可能暫停）

### 替身策略類型

| 策略名 | 說明 | 範例 |
|--------|------|------|
| 法器替換 | 換一種工具做同樣的事 | `ripgrep` 不可用 → 退為 `grep` |
| 降級執行 | 降低品質要求但完成任務 | 完整分析 → 抽樣分析 |
| 拆分攻克 | 將大任務拆小重試 | 整目錄處理失敗 → 逐檔處理 |
| 繞道而行 | 用完全不同的方法 | API 呼叫失敗 → 從快取讀取 |
| 人工呈報 | 放棄自動化，將問題整理後交給法師 | 記錄失敗原因 + 手動步驟清單 |

### 大法主敕令片段（含替身）

```
敕令 [程式碼分析神將]：
  主策略：使用 AST parser 分析 [語言] 程式碼
  替身策略：
    第一替身：使用正則表達式搜尋
      觸發條件：AST parser 執行失敗或不支援此語言
    第二替身：逐行文字搜尋
      觸發條件：正則搜尋也失敗
    末路策略：將檔案清冊與錯誤訊息呈報法師
      觸發條件：所有自動策略皆失敗

每次降級須記錄：
- 主策略失敗原因
- 降級至哪一層
- 降級後的功德品質評估（例如「覆蓋率從 100% 降至 60%」）
```

### 替身符籙寫法

為主將和替身各寫獨立符籙，但在大法主符中定義觸發關係：

```
神將配置：
  主將：
    法號：AST分析真人
    符籙：[主將符籙內容]
    圓滿判定：功德簿存在且非空

  第一替身：
    法號：正則搜索道兵
    符籙：[替身符籙內容]
    觸發條件：主將功德簿不存在 或 主將回報天劫
    額外供物：主將的天劫記錄（讓替身知道哪裡出了問題）

  末路替身：
    法號：呈報道兵
    符籙：收集所有天劫記錄，整理為法師可讀的報告
    觸發條件：第一替身也渡劫失敗
```

### 替身設計戒律

- **替身不可比主將更複雜**——越往後越簡單，這是降級不是升級
- **每次降級都要記錄**——最終功德簿必須說明用了哪一層策略
- **替身壇場獨立**——不要覆蓋主將的半成品，各存各的
- **最多三層替身**——主將 + 兩替身 + 末路呈報，再多則應重新設計任務
