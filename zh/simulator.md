# 演法壇 ── Ritual Simulator / Dry Run

正式開壇前，先以幻象演練。用 mock 資料走一遍完整科儀，驗證壇位銜接、法物流轉、渡劫預案，不消耗真法力。

---

## 何時演法

- 科儀涉及 3 位以上神將
- 科儀中有巢狀結構
- 法事涉及生產環境或不可逆操作
- 首次使用新的符籙或科儀陣式
- 法師想在正式行法前確認流程無誤

---

## 演法流程

### 第一步：備幻象供物（Mock Input）

為科儀的初始輸入準備簡化的 mock 資料：

```
演法供物：
- 類型：[與真實供物相同的格式]
- 內容：[簡化版——3-5 筆代表性資料即可]
- 存放：/tmp/rehearsal/[科儀名]/input/

範例：
若真實供物是 1000 個 Python 檔案，演法供物只需 3 個：
- mock_simple.py（正常案例）
- mock_edge.py（邊界案例）
- mock_error.py（預期會觸發天劫的案例）
```

### 第二步：標記演法模式

在大法主敕令中加入演法標記：

```
⚙️ 演法模式啟動

本次為演法（dry run），遵循以下規則：
1. 使用 /tmp/rehearsal/ 為壇場根目錄
2. 不得存取真實資料來源
3. 不得執行任何外部 API 呼叫（模擬回應即可）
4. 不得推送任何變更到遠端
5. 所有「寫入」操作只寫到演法壇場
6. 每位道兵在功德簿中標注「[演法]」前綴

演法目標：
- 驗證科儀流程是否能走完
- 驗證壇位之間的法物格式是否對接
- 驗證渡劫預案是否能觸發
- 測量大致耗時與法力消耗
```

### 第三步：執行演法

按正式科儀流程執行，但使用演法供物與演法壇場。

### 第四步：演法覆盤

演法結束後，產出覆盤報告：

```markdown
# 演法覆盤報告

## 科儀概要
- 科儀名：[名]
- 科儀類型：[類型]
- 壇位數：[N]
- 演法時間：[duration]

## 流程驗證

| 壇位 | 道兵 | 狀態 | 備註 |
|------|------|------|------|
| 1 | [法號] | ✅ 通過 | |
| 2 | [法號] | ✅ 通過 | |
| 3 | [法號] | ⚠️ 警告 | [法物格式略有不符] |
| 4 | [法號] | ❌ 失敗 | [渡劫預案未覆蓋此情況] |

## 法物流轉驗證

| 來源壇 → 目標壇 | 法物格式 | 對接結果 |
|----------------|---------|---------|
| 壇1 → 壇2 | JSON | ✅ 格式正確，欄位齊全 |
| 壇2 → 壇3 | Markdown | ⚠️ 缺少 `summary` 欄位 |

## 渡劫預案驗證
- TC-mock_error.py 觸發天劫：[是否被正確處理]
- 替身道兵觸發：[是否按預期降級]
- 輪值關口觸發：[是否正確暫停]

## 法力估算
- 演法消耗：[X]K tokens（含 mock 資料）
- 推估正式消耗：[Y]K tokens（按資料量比例放大）

## 問題清冊
1. [問題描述] → [建議修正]
2. [問題描述] → [建議修正]

## 演法結論
- ✅ 可正式開壇（流程無阻）
- ⚠️ 修正後可開壇（有小問題需先處理）
- ❌ 不可開壇（有關鍵問題需重新設計）
```

---

## 演法 vs 正式對照

| 項目 | 演法模式 | 正式模式 |
|------|---------|---------|
| 壇場 | /tmp/rehearsal/ | 真實目錄 |
| 供物 | Mock 資料（3-5 筆） | 真實資料 |
| API 呼叫 | 模擬回應 | 真實呼叫 |
| 寫入操作 | 只寫演法壇場 | 寫真實目標 |
| 外部推送 | 禁止 | 按敕令執行 |
| 功德簿前綴 | [演法] | （無） |
| 法力消耗 | 約正式的 10-20% | 100% |

---

## 局部演法

不必每次都走完整科儀。可針對特定壇位或銜接點做局部演法：

```
局部演法：壇位 2 → 壇位 3 銜接測試

目的：驗證壇位 2 的輸出格式能否被壇位 3 正確讀取。

步驟：
1. 手動準備壇位 2 的模擬功德（或使用上次演法的產物）
2. 只敕令壇位 3 的道兵，以模擬功德為供物
3. 確認壇位 3 能正確讀取並處理

適用場景：
- 修改了某位道兵的符籙後，只需驗證與上下壇的銜接
- 新增壇位時，驗證新壇與既有壇的兼容性
```
